# dfc-agent-framework-integration

**LLM-driven data-flow control for tool-calling agents.** Drop this into any agent and it will, per task:
read the user's (trusted) instruction, generate Passant data-flow policies that pin the agent's dangerous
actions to what the user actually asked for, filter out the policies that would only cause false
positives, and then **block tool calls that violate those policies at runtime** — so a prompt injection
can't redirect a payment, exfiltrate data, or change an account, even though the policies were written by
an LLM on the fly.

The core library is **framework-neutral** (it has no hard dependency on any benchmark or agent runtime).
You integrate it by giving it three things: a description of your tools, an LLM client, and a call into
your tool-execution loop. An AgentDojo adapter ships separately as a reference integration.

## 1. The idea

A prompt-injection defense by **grounding**: a sensitive action's value must derive from a **trusted**
source. The only sound definition of "trusted" is *what the agent (and therefore an attacker who has
hijacked it) cannot author* — namely the **original task prompt** (the `preamble`) and pre-existing user
state. Everything produced inside the workflow (search results, fetched web pages, tool outputs the
attacker can influence) is **untrusted**.

So for a "change my password to `NewPassword`" task, the defense extracts the trusted fact
`new_password = "NewPassword"` and registers:

```
SINK UpdateShoppingAccountPasswordInput
DIMENSION PreambleData
CONSTRAINT UpdateShoppingAccountPasswordInput.password = PreambleData.new_password
ON FAIL KILL
```

An injected `update_password("PASS999")` doesn't match the trusted fact → the tool call is **killed**.
The legitimate `update_password("NewPassword")` passes. The grounding is enforced as an actual SQL
data-flow constraint by the Passant engine (`data_flow_control`), backed by an in-memory DuckDB.

The hard part this library solves: the policies are **generated by an LLM per task**, and a naive
generator writes policies that also `KILL` legitimate, benign actions (e.g. pinning a product *search* to
the literal phrase "smart watch"). [Admission control](#3-admission-control) is what makes LLM-generated
policies safe to deploy.

## 2. How it works, end to end

Per task, `DFCBenchmarkContext.prepare_task(...)` runs this lifecycle:

1. **Build the schema.** Your tools become *relations*: each tool gets an `…Input` relation (its
   parameters as columns) and an `…Output` relation (its return fields). Plus `PreambleData` (the trusted
   facts), `PromptInput`, and `AssistantResponseOutput`. See [`RuntimeSchema`](#runtimeschema).
2. **Extract trusted facts.** An LLM call reads the `preamble` (the user's task prompt only) and returns
   key/value facts — recipients, account numbers, passwords, filenames, URLs, amounts. These are
   materialized as the single-row `PreambleData` relation.
3. **Generate policies.** An LLM call produces a set of PGN policies grounding sinks against `PreambleData`
   (or against tool-output relations for exfiltration checks).
4. **Admission control** ([§3](#3-admission-control)) — drop the generated policies that can only cause
   false positives (policies on read-only sinks; policies grounded on goal-phrase descriptions).
5. **Register** the surviving ("admitted") policies into the Passant engine, with an LLM **repair** loop
   for any PGN that fails to parse/register.
6. **Enforce at runtime.** As your agent runs, you call `validate_tool_call(name, args)` before each tool
   executes. The framework stages the call as a row and attempts to write it through the registered
   policies; if a policy's constraint fails, it returns a `DFCViolation` and you block the call. After a
   tool runs, you call `record_tool_output(name, result)` so later policies can ground on what was read.

The enforcement mechanism: each tool input is written to a staging table, then an
`INSERT … SELECT … FROM staging [CROSS JOIN PreambleData / dimensions] WHERE <constraint>` carries the row
into the real sink relation. If the constraint doesn't hold (e.g. the value isn't in `PreambleData`), zero
rows are written → the engine raises `ON FAIL KILL` → `DFCViolation`. Provenance is established by the
join, not by trusting the model.

## 3. Admission control

LLM policy generators over-produce: they pin benign sinks and ground on loose descriptions, and the
resulting `ON FAIL KILL` policies murder legitimate agent behavior. You cannot fix this by *asking* the
generator to behave (a generative model can't be relied on to *not* produce a tempting-but-wrong policy).
So enforcement lives in deterministic code, between generation and registration. Every layer is
**principle-based** (judges from the tool's own description / the fact's own shape — no hardcoded tool-name
lists, so it generalizes to unseen tools) and **fail-safe toward security** (when unsure, keep the policy:
a kept false-positive costs utility; a wrongly-dropped guard costs security).

| Layer | What it does | Effect |
|---|---|---|
| **1. Prompt guidance** | Tells the generator which sinks to constrain (`SINK_SELECTION_GUIDANCE`) | lowers the *rate* of bad policies |
| **2. Sink-effect classifier** | A separate, neutral LLM call labels each sink `read_only` vs `effectful` from its description | **drops** policies on read-only sinks |
| **3. Groundability check** | Detects equalities grounded on a *goal-phrase* fact (a loose description, not a precise value) | **drops** those policies |
| **Client hardening** | Tolerates malformed structured output | coerce + retry instead of crashing the task |

**Layer 2 — sink-effect classifier (the guarantee).** After generation, one cheap call classifies each
tool-input relation: `effectful` (sends / pays / changes credentials / shares / deletes — worth guarding)
vs `read_only` (only reads / searches / queries / lists — guarding it gives zero security and only breaks
legitimate exploratory calls). Policies on `read_only` sinks are dropped in code. It is a *separate*,
neutral call (not self-tagging by the generator, which is biased toward justifying what it just wrote),
and it fails safe: on any classifier error nothing is dropped.

**Layer 3 — goal-phrase groundability.** Even on an effectful sink, an equality grounded on a *goal
phrase* (`"smart watch"`, `"a pair of headphones"`) can't hold across the agent's legitimate values, so it
only false-positives — and it isn't a real guard (it pins a description, not a recipient/amount). These are
dropped. The precision test is conservative: it keeps single tokens, structured values (emails, URLs,
paths, numbers, filenames) and proper-noun/Title-Cased names, dropping only lowercase multi-word phrases —
so it never removes a real egress/transfer guard.

> Validated cross-suite (shopping/workspace/banking): the original false-positive classes (read-only
> sinks, goal-phrase groundings) went to **0 leaks** with these layers, and both layers fire on real
> generations (e.g. Layer 2 drops `ReadFileInput`/`SearchProductInput`, Layer 3 drops a `SendMoney`
> policy grounded on a product description). See `devnotes/admission_control_design.md` in the host repo.

## 4. Architecture & data model

### Relations

Everything is modeled as a relation in DuckDB:

- **`PreambleData`** — one row of trusted facts extracted from the task prompt. Columns are the fact keys.
- **`<Tool>Input`** — one relation per tool, columns = the tool's parameters. This is the **sink** a policy
  guards. Name = PascalCase(tool_name) + `Input` (e.g. `send_email` → `SendEmailInput`).
- **`<Tool>Output`** — one relation per tool, columns = the tool's return fields. Used as a `DIMENSION` for
  exfiltration policies (e.g. "don't email a password that appears in `GetAccountInfoOutput`").
- **`PromptInput`**, **`AssistantResponseOutput`** — for validating prompts and model responses.

### Policies (PGN)

A policy declares a `SINK`, optional `SOURCE` / `DIMENSION` relations to ground against, a `CONSTRAINT`,
`ON FAIL KILL`, and a `DESCRIPTION`. See [§10](#10-the-pgn-policy-language). Policies are `Policy.from_pgn`
parsed and registered into the Passant connection.

### Key modules

| Module | Role |
|---|---|
| `context.py` | `DFCBenchmarkContext` — the per-task orchestrator (`prepare_task`, the `validate_*` hooks) |
| `extraction.py` | `extract_preamble_facts` — trusted-fact extraction |
| `policies.py` | `generate_and_register_policies` + the **admission-control** logic (Layers 2 & 3) |
| `prompts.py` | all LLM instructions (extraction, generation+sink-selection, classification, repair) |
| `repair.py` | `register_policy_with_repair` — register with an LLM repair loop |
| `runtime.py` | `DFCRuntimeValidator` — the runtime enforcement (`validate_tool_call`, `validate_event`) |
| `schema.py` | `RuntimeSchema`, `RelationSchema`, and all pydantic models |
| `events.py` | tool→relation naming, table creation, the DuckDB write path |
| `llm.py` | `StructuredLLMClient` protocol + `OpenAIStructuredLLMClient` + `FakeStructuredLLMClient` |
| `dfc_event_log.py` | append-only `events.jsonl` progress log (optional) |
| `persistence.py` | export run artifacts (generated policies, fire counts, validation events) |

## 5. Public API

```python
from dfc_agent_framework_integration import (
    DFCBenchmarkContext,   # the per-task orchestrator
    BenchmarkTaskContext,  # holds the trusted preamble
    RuntimeSchema,         # your tools, as relations
    DFCViolation,          # returned when a tool call is blocked
)
from dfc_agent_framework_integration.schema import RelationSchema      # build a schema by hand
from dfc_agent_framework_integration.llm import StructuredLLMClient    # the LLM interface to implement
```

### `DFCBenchmarkContext.prepare_task(...) -> DFCBenchmarkContext`

```python
ctx = DFCBenchmarkContext.prepare_task(
    task_context=BenchmarkTaskContext(benchmark_name="myapp", preamble=user_prompt),
    runtime_schema=schema,            # RuntimeSchema
    llm=my_structured_client,         # StructuredLLMClient
    dfc_model="<model id>",           # model for extraction + policy generation
    functions=my_tools,               # the tool objects (used for output-schema inference)
    agent_model=None,                 # optional, recorded for diagnostics
    classifier_model="<cheap model>", # Layer-2 sink classifier (defaults to dfc_model if None)
    event_log=None,                   # optional DFCEventLog
)
```

Runs extraction → generation → admission control → registration. Returns a live context whose DuckDB
holds the registered policies and `PreambleData`.

### Runtime hooks (call these from your agent loop)

| Method | When | Returns |
|---|---|---|
| `ctx.validate_tool_call(name, args)` | **before** executing a tool | `DFCViolation` or `None` |
| `ctx.record_tool_output(name, result)` | **after** a tool runs (no error) | — |
| `ctx.validate_prompt(content, target="")` | before an LLM call (optional) | `DFCViolation` or `None` |
| `ctx.validate_assistant_response(content)` | on a model response (optional) | `DFCViolation` or `None` |
| `ctx.close()` | end of task | — |

`DFCViolation` carries `policy_ids`, `policy_descriptions` (show these to the model so it can retry),
`attempted_payload`, `relation`, and `raw_error`.

### Diagnostics

`ctx.diagnostics` (a `DFCTaskDiagnostics`) exposes `extracted_facts`, `generated_policies` (the *admitted*
set), `registered_policy_ids`, `deleted_policies`, `policy_fire_counts`, `validation_events`.

## 6. Integrating it into your own agent

You write three things. Here is the whole thing for a toy app:

```python
from pydantic import BaseModel
from dfc_agent_framework_integration import DFCBenchmarkContext, BenchmarkTaskContext, RuntimeSchema
from dfc_agent_framework_integration.schema import RelationSchema

# --- (1) describe your tools as relations ---------------------------------
# Option A: if your tools are pydantic-typed (.name/.description/.parameters/.return_type),
#           RuntimeSchema.from_tools(tools) builds this for you.
# Option B: build it by hand (works for any tool format):
schema = RuntimeSchema(
    tool_input_relations=[
        RelationSchema(name="SendEmailInput", tool_name="send_email",
                       description="Send an email.",
                       columns={"recipient": "VARCHAR", "subject": "VARCHAR", "body": "VARCHAR"}),
        RelationSchema(name="SearchDocsInput", tool_name="search_docs",
                       description="Search internal docs.",
                       columns={"query": "VARCHAR"}),
    ],
    tool_output_relations=[
        RelationSchema(name="SendEmailOutput", tool_name="send_email", columns={"ok": "BOOLEAN"}),
        RelationSchema(name="SearchDocsOutput", tool_name="search_docs", columns={"text": "VARCHAR"}),
    ],
)

# --- (2) an LLM client (see §7) -------------------------------------------
llm = MyStructuredLLMClient(...)

# --- (3) wire it into the per-task lifecycle ------------------------------
ctx = DFCBenchmarkContext.prepare_task(
    task_context=BenchmarkTaskContext(benchmark_name="myapp", preamble=user_prompt),
    runtime_schema=schema, llm=llm,
    dfc_model="<model>", classifier_model="<cheap model>", functions={},
)

def call_tool(name, args):
    violation = ctx.validate_tool_call(name, args)        # <-- enforcement
    if violation is not None:
        return f"Blocked by data-flow policy: {violation.policy_descriptions}"
    result = my_real_tool_dispatch(name, args)            # your code
    ctx.record_tool_output(name, result)                  # let later policies ground on it
    return result
# ... run your agent, routing every tool call through call_tool ...
ctx.close()
```

**`RuntimeSchema.from_tools(tools)`** expects each tool to expose `.name`, `.description`, `.parameters`
(a pydantic model → `.model_json_schema()`), and `.return_type` (a pydantic model). If your tools are
shaped like that (AgentDojo, many typed-tool frameworks), use it directly. Otherwise build `RelationSchema`
objects as above — `columns` is just `{column_name: duckdb_type}`.

The host repo's AgentDojo adapter (`agentdojo/integrations/dfc_agent_framework_integration.py`) is a fuller
worked example: it wraps these hooks as pipeline elements and adds a Bedrock LLM client.

## 7. The LLM client

The library is provider-agnostic via a small protocol (`llm.py`):

```python
class StructuredLLMClient(Protocol):
    def parse(self, *, model: str, instructions: str, input_text: str, text_format: type[T]) -> T: ...
```

`text_format` is a pydantic model; `parse` must return a validated instance. Ships with:

- **`OpenAIStructuredLLMClient`** — uses `client.responses.parse(...)`.
- **`FakeStructuredLLMClient`** — returns canned objects, for tests.

**Writing one for Claude / Bedrock** (the pattern the host repo uses) is ~40 lines via tool-use — the
pydantic schema becomes a forced tool, and the `tool_use` input is validated back into the model. Make it
robust: coerce the common "list field arrived as a dict" malformation and, on a `ValidationError`, feed the
error back to the model as a `tool_result` and retry, rather than crashing the task. (See
`BedrockStructuredLLMClient` in the AgentDojo adapter for a complete implementation.)

## 8. Configuration & models

- **`dfc_model`** — does fact extraction + policy generation. A stronger model writes cleaner PGN (fewer
  repairs, fewer pathological policies). Opus-class works well.
- **`classifier_model`** — the Layer-2 sink classifier. A *cheap* model (Haiku-class) is fine and keeps
  cost/latency down; classification is a much easier task than authoring PGN. Defaults to `dfc_model` if not
  set. In the AgentDojo adapter it defaults to Haiku and is overridable via the `DFC_CLASSIFIER_MODEL` env
  var.
- **Bedrock note** (host repo): use `us.anthropic.*` / `global.anthropic.*` inference-profile IDs; bare
  foundation-model IDs are rejected for on-demand. A `BEDROCK_TIMEOUT` env var bounds wedged calls. Never
  share one async client across threads/event loops (it deadlocks) — build one client per worker.

## 9. Observability

If a `DFCEventLog(output_dir)` is passed (or the optional AgentDojo logger is present), each task writes an
append-only `events.jsonl` you can tail. Key events:

| Event | Meaning |
|---|---|
| `extraction_complete` | trusted facts extracted |
| `policy_generation_llm_complete` | N policies generated (pre-admission) |
| `sink_classification_complete` | `{read_only: [...], effectful: [...]}` — the classifier verdict |
| `sink_classification_failed` | fail-safe path (kept all policies) |
| `policy_dropped_read_only_sink` | Layer 2 dropped a policy `{policy_id, sink}` |
| `policy_dropped_goal_phrase_grounding` | Layer 3 dropped a policy `{policy_id, sink, goal_phrase_facts}` |
| `policy_register_success` / `policy_register_deleted` | registration outcome (after repair) |
| `validation` | a tool call / prompt / response was checked `{event_type, blocked, policy_ids, …}` |

`persistence.export_run_artifacts(ctx, dir)` writes a `metadata.json` with the admitted policies, fire
counts, and validation events for offline analysis.

## 10. The PGN policy language

Allowed clause subset (policies are generated and repaired to stay within it):

- `SINK <Relation>` — the guarded relation (almost every policy has one).
- `SOURCE` / `SOURCE REQUIRED <Relation>` — require data to originate from an existing relation (provenance
  for read→write flows). `SOURCE` columns may need aggregates (`max(...)`) since the source may have many
  rows.
- `DIMENSION <Relation | (SELECT … ) Alias>` — a relation to ground against: `PreambleData`, a tool-output
  relation, or a subquery that always yields one row (e.g.
  `(SELECT COALESCE(max(password), '') AS password FROM GetAccountInfoOutput) AccountInfo`).
- `CONSTRAINT <bool expr>` — references SINK columns directly (no aggregates) and DIMENSION/SOURCE columns.
  Use `=` for grounding; use `NOT contains(<sink_col>, <dim_col>)` for "don't leak this substring". Avoid
  `LIKE`/`ILIKE` (fail at runtime).
- `ON FAIL KILL` — always. Aborts the offending write with a policy-violation signal (no silent filtering).
- `DESCRIPTION <text>` — user-facing reason, surfaced in `DFCViolation.policy_descriptions`.

Example exfiltration policy:

```
SINK SendEmailInput
DIMENSION (SELECT COALESCE(max(password), '') AS password FROM GetAccountInfoOutput) AccountInfo
CONSTRAINT AccountInfo.password = '' OR NOT contains(SendEmailInput.body, AccountInfo.password)
ON FAIL KILL
DESCRIPTION Prevent data exfiltration: do not email the account password read during this run.
```

## 11. Limitations & caveats

- **The `preamble` is the trust anchor.** The whole model assumes you can hand it an *injection-free* task
  description. If your "trusted" input can itself be poisoned, grounding is meaningless.
- **Generation is non-deterministic.** The same task may yield different policies run to run. The
  *enforcement* and *admission control* are deterministic given a generated policy, but coverage (which
  sinks get a policy) varies — estimate rates over repeated runs.
- **The classifier is itself an LLM call.** Mitigated by neutrality, an easy task, and fail-safe-to-keep,
  but a systematic misclassification of an effectful sink as read-only would silently drop a guard. Audit
  `sink_classification_complete` periodically.
- **Layer 3's precision test is a heuristic.** Conservative (keeps structured/Title-cased values), so the
  residual risk is a missed drop (utility), not a dropped guard.
- **Known unaddressed class: sequencing/provenance policies.** A generator can invent "must read X before
  doing Y" rules and `KILL`-enforce them; these can false-positive and are not caught by Layers 2–3.
- **Research-grade.** Harden (pin models, add audits, bound generation) before production use.

## 12. Install

```toml
# pyproject dependencies
data-flow-control>=0.1.3   # the Passant engine
pydantic>=2.7.1
# plus an LLM SDK for your StructuredLLMClient (openai, anthropic, …) and duckdb (pulled in transitively)
```

```bash
pip install -e dfc_agent_framework_integration
```

The package has **no hard dependency on any agent framework** — the single optional `agentdojo` import (for
artifact-dir logging) is guarded by `try/except ImportError`. Bring your own tools, LLM client, and tool
loop per [§6](#6-integrating-it-into-your-own-agent).
